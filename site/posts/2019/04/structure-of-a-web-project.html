WEB项目的结构

最近自己写了两个项目都需要前端和后端一起开发,由于以前只做后端,所以没有认真想过如果一个人做完整个项目需要考虑什么问题,

最近的两个项目让我遇到了不少问题,当然每个人的实际情况不一样,本人遇到的这写问题可以通过调整项目结构来解决,接下来进入正题.

开发工具

Docker, Nginx, MySQL 和 Webpack.

跨域问题

只要是前后端分离都会遇到跨域问题的,跨域问题其实没什么好说的,目前常见的有两种方式可以解决这个问题:

  1. 利用 Nginx 这种后端程序做反向代理;
  2. 利用 W3C 的 CORS 机制.

还有一些其它的什么 JSONP ,这些本人都不考虑,相对于前两种方法,这些"其它"多少有些缺陷.

目前本人见到比较多的前后端协作方式就是:代码写完然后 push 代码到同一个服务器特定位置上.这种方式就是第一种方案,各个端的开发人员一般都会在本地调试好再 push (也就是第一种方法).

但是这种协作方式会对调试造成一定麻烦: 对于需要从前后端调试的功能要求每次改一下就 PUSH 一下.不过还好单人开发没有这种问题.

本人目前开发前端是使用了自己的 Webpack 配置,开发时候 Webpack 的 devserver 会启动一个后端服务,这套配置可以在修改样式和 JavaScript 文件的时候自动刷新(hot reload),这些变更都是在内存上发生的并且不会写入硬盘.

而后端则是又单独一个服务,虽然也有跨域问题,但是我可以在开发过程中采取第二种方式, 前期只要针对后端配置一下 Nginx ,对于前端则需要定义一个服务器地址的全局变量,到部署的时候再把该变量改会""空字符串,针对前端文件配置一下 Nginx,并且去掉 CORS 需要的首部.

这种方式优点在于减少开发过程中编码以外的操作,并且一定程度上保证了开发处于生产环境(一般来说,后端配置会麻烦于前端).

要注意的是, CORS 首部是只在 Nginx 上添加的,避免最后关头修改代码的发生.
